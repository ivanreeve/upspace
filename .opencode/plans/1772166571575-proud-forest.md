# Admin Features Audit Plan

## Audit Summary

After a thorough review of all admin-related code (12 API routes, 10 page components, 6 hooks, auth guards, middleware, and validation schemas), here are the findings organized by severity.

---

## Bugs Found (Must Fix)

### Bug 1: Enable route incorrectly sets `cancelled_at` to `now`
**File:** `src/app/api/v1/admin/users/[user_id]/enable/route.ts:80`
**Issue:** When re-enabling a deactivated user, `cancelled_at` is set to `now` instead of `null`. This timestamps the user as "cancelled" at the moment they are being re-enabled, which is semantically wrong and could affect downstream logic that checks `cancelled_at`.
**Fix:** Change `cancelled_at: now` to `cancelled_at: null`.

### Bug 2: Deactivation approval raw SQL runs outside transaction
**File:** `src/app/api/v1/admin/deactivation-requests/[request_id]/route.ts:100-106`
**Issue:** For `deactivate` type requests, the `$executeRaw` that updates `auth.users.raw_user_meta_data` runs **before** the `$transaction(updates)` at line 123. If the transaction fails after the raw SQL succeeds, `auth.users` metadata will be out of sync with the `user` table. The disable/enable routes correctly put their raw SQL inside the transaction.
**Fix:** Move the `$executeRaw` into the `updates` array so it runs inside the transaction.

### Bug 3: Verification PATCH allows re-approving/re-rejecting
**File:** `src/app/api/v1/admin/verifications/[verification_id]/route.ts:126-148`
**Issue:** The route checks that the verification exists but does NOT verify that `status === 'in_review'` before updating. An admin could approve an already-approved verification or reject an already-rejected one, overwriting timestamps and reasons.
**Fix:** Add a status check: `if (verification.status !== 'in_review') return 400 "This verification has already been processed."`.

### Bug 4: No server-side protection against disabling/enabling admin accounts
**Files:**
- `src/app/api/v1/admin/users/[user_id]/disable/route.ts`
- `src/app/api/v1/admin/users/[user_id]/enable/route.ts`
**Issue:** The UI prevents toggling admin accounts (`user.role !== 'admin'` at `AdminUsersPage.tsx:304`), but neither API route checks the target user's role. A direct API call could disable/enable another admin's account.
**Fix:** After fetching `targetUser`, check `if (targetUser.role === 'admin') return 403 "Admin accounts cannot be modified."`. Also need to add `role: true` to the `select` clause.

### Bug 5: Deactivation request type not displayed in admin UI
**File:** `src/components/pages/Admin/AdminDeactivationRequestsPage.tsx`
**Issue:** The API returns a `type` field (`deactivate` or `delete`) for each request, but it's missing from the client type (`useAdminDeactivationRequests.ts:9-25`) and not displayed anywhere in the UI. This is critical context for the admin since:
- `deactivate` = immediate account deactivation
- `delete` = 30-day pending deletion deadline

An admin approving a request without knowing the type could unknowingly trigger account deletion.
**Fix:** Add `type` to the `DeactivationRequest` client type and display it as a badge/label in both the table and card layouts.

### Bug 6: Client-side sort conflicts with cursor pagination in deactivation requests
**File:** `src/hooks/api/useAdminDeactivationRequests.ts:92-93`
**Issue:** The API returns data sorted `created_at: 'asc'` (oldest first for cursor pagination), but the client re-sorts to `desc` (newest first). This creates a confusing UX where items within a page are newest-first, but paging forward fetches the next batch of oldest items. The sort order within each page contradicts the page progression order.
**Fix:** Either remove the client-side sort (accept server order) or change the API to sort `desc` and update cursor logic accordingly.

### Bug 7: Enable action visible for deleted users in UI
**File:** `src/components/pages/Admin/AdminUsersPage.tsx:302`
**Issue:** `const showEnableAction = user.status !== 'active'` means the "Enable account" button appears for `deleted` users. While the API returns a 400 error for deleted accounts, the UI shouldn't offer the action.
**Fix:** Change to `const showEnableAction = user.status !== 'active' && user.status !== 'deleted'`.

## Minor Issues (Should Fix)

### Issue 8: Deactivation rejection maxLength mismatch
**Files:**
- `AdminDeactivationRequestsPage.tsx:552` — textarea `maxLength={400}`
- `deactivation-requests/[request_id]/route.ts:15` — schema allows `max(1000)`
**Fix:** Align both to 1000 characters.

### Issue 9: Space visibility mutation cache invalidation incomplete
**File:** `src/hooks/api/useAdminVerifications.ts:228-230`
**Issue:** `useAdminSpaceVisibilityMutation` only invalidates `adminVerificationKeys` on success. When used from the Spaces page, the page manually invalidates `adminSpacesKeys` (line 142 of `AdminSpacesPage.tsx`), but when used from the Unpublish Requests page for republishing, neither `adminSpacesKeys` nor `adminUnpublishRequestKeys` are invalidated by the mutation itself.
**Fix:** The mutation's `onSuccess` should also invalidate `adminSpacesKeys` and `adminUnpublishRequestKeys`.

---

## Observations (No Action Required)

- **Dashboard client metrics**: Only counts `role: 'customer'` users. Partner and admin users excluded from "Clients" metrics. This appears intentional.
- **No admin seed data**: Admin users must be manually created by updating the DB. This is noted but likely intentional for security.
- **Middleware performance**: Two internal API calls (`sync-profile` + `profile`) on every request add latency. Not admin-specific.
- **`parseErrorMessage` duplicated**: 6 identical copies across hook files. Could be extracted to a shared utility. Low priority.
